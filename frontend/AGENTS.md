# AGENTS.md (Frontend Subdirectory)

> 이 문서는 상위 AGENTS.md를 보완하며, `frontend/` 하위에만 적용됩니다.

---

## 1) 적용 범위
- 적용 대상: `frontend/` 및 모든 하위 디렉터리
- 우선순위: 상위 AGENTS와 충돌 시 이 파일 규칙 우선

---

## 2) 프론트엔드 영역 책임
- UI 컴포넌트, 페이지, 라우팅, 상태관리, API 연동(클라이언트 측), 스타일
- 프론트 변경은 가능한 한 프론트 레이어에서 해결
- 백엔드 계약(API 스펙) 변경 필요 시 반드시 명시

---

## 3) 구현 원칙
1. **사용자 경험 안정성**
   - 깨지는 UI/라우팅/폼 동작을 만들지 않는다.
2. **접근성 기본 준수**
   - 버튼/링크 역할 구분, 레이블, 키보드 접근성 고려
3. **재사용 우선**
   - 중복 컴포넌트 생성보다 기존 공용 컴포넌트 활용
4. **성능 고려**
   - 불필요한 리렌더, 과도한 번들 증가 방지
5. **에러 처리 일관성**
   - 사용자 메시지(친절/간결)와 내부 로그(상세) 분리

---

## 4) 디렉터리/파일 배치 규칙 (필요 시 팀 구조에 맞게 수정)
- 페이지/라우트: `frontend/src/pages` 또는 `frontend/src/app`
- 공통 컴포넌트: `frontend/src/components`
- 기능 단위 모듈: `frontend/src/features/<feature-name>`
- API 클라이언트: `frontend/src/services` 또는 `frontend/src/api`
- 스타일/테마: `frontend/src/styles` 또는 컴포넌트 인접 배치
- 테스트:
  - 단위 테스트: 소스 인접(`*.test.*`) 또는 `__tests__/`
  - E2E: `frontend/e2e` (사용 시)

---

## 5) 네이밍/코딩 규칙
- 컴포넌트: `PascalCase`
- 함수/변수: `camelCase`
- 상수: `UPPER_SNAKE_CASE`
- CSS 클래스 네이밍은 프로젝트 기존 규칙 준수 (BEM, utility, CSS-in-JS 등)
- 타입/스키마는 가능한 명시적으로 관리(TypeScript 사용 시)

---

## 6) UI 변경 시 체크리스트
- [ ] 레이아웃 깨짐 없음 (주요 해상도)
- [ ] 폼 유효성/에러 메시지 정상
- [ ] 로딩/에러/빈 상태 처리
- [ ] 접근성(포커스, aria, semantic) 기본 확인
- [ ] 다크모드/테마 영향(해당 시)

---

## 7) 테스트 규칙 (Frontend)
가능하면 아래 순서로 실행:
1. 린트
2. 단위 테스트
3. 빌드
4. (있다면) E2E/스토리북 검증

### 권장 명령 (프로젝트에 맞는 것만 사용)
- `npm run lint`
- `npm run test`
- `npm run build`
- `npm run test:e2e`
- `pnpm lint && pnpm test && pnpm build`
- `yarn lint && yarn test && yarn build`

> 실제 패키지 매니저(npm/pnpm/yarn)와 스크립트 명은 `package.json` 기준으로 확정합니다.

---

## 8) API 연동 규칙
- API 요청/응답 타입을 명확히 유지
- 에러 코드는 사용자 메시지와 내부 진단 로그를 분리
- 재시도/타임아웃/취소 정책은 기존 유틸 패턴 재사용
- 스펙 변경 시 문서/목(Mock)/테스트 동시 업데이트

---

## 9) 리뷰 시 중점 확인
1. 사용자 흐름이 실제로 자연스러운가?
2. 예외/경계값 처리 누락은 없는가?
3. 코드 중복 없이 유지보수 가능한가?
4. 테스트가 변경 의도를 충분히 보장하는가?

---

## 10) 금지 사항
- 임시 콘솔 로그/디버거 코드 커밋
- 무분별한 전역 스타일 오염
- 요청 범위를 벗어난 대규모 UI 리뉴얼
- 근거 없는 의존성 추가
