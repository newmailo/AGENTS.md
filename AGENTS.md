# AGENTS.md (Repository Root)

## 목적
이 저장소에서 작업하는 에이전트/기여자에게 공통 작업 규칙을 제공합니다.

## 적용 범위
- 이 파일이 있는 디렉터리 이하 전체 트리에 적용됩니다.
- 더 하위 디렉터리에 별도 `AGENTS.md`가 있으면, 하위 파일 규칙이 우선합니다.

## 기본 원칙
1. 변경은 최소 단위로 수행하고, 관련 없는 수정은 하지 않습니다.
2. 기존 코드 스타일/패턴을 우선 따릅니다.
3. 큰 구조 변경 전에는 간단한 계획(무엇/왜/영향)을 먼저 제시합니다.
4. 민감정보(API 키, 토큰, 비밀번호)는 절대 커밋하지 않습니다.
5. 주석/문서/커밋 메시지는 팀 규칙(언어, 포맷)에 맞춥니다.

## 작업 절차
1. 현재 상태 파악: 디렉터리 구조, 실행/테스트 방법 확인
2. 변경 계획 수립: 파일별 변경 포인트 정리
3. 구현: 필요한 파일만 수정
4. 검증: lint/test/build 실행
5. 요약: 변경 사항, 검증 결과, 리스크/추가 과제 공유

## 코드 스타일
- 포매터/린터가 있으면 해당 규칙을 최우선으로 준수합니다.
- 불필요한 리팩터링/네이밍 변경은 지양합니다.
- import 정렬, 타입/에러 처리, 로깅 패턴은 기존 관례를 따릅니다.

## 테스트/검증
- 변경 범위를 커버하는 최소 테스트를 실행합니다.
- 가능하면 아래 순서로 점검합니다:
  - lint
  - unit test
  - build
  - (필요 시) integration/e2e
- 실패 시 원인/영향/우회 방법을 기록합니다.

## 커밋 규칙
- 하나의 논리적 변경을 하나의 커밋으로 만듭니다.
- 커밋 메시지 예시:
  - `feat: add user profile endpoint`
  - `fix: handle null response in payment flow`
  - `chore: update lint config`

## PR 규칙
- PR 본문에 아래 항목을 포함합니다:
  1. 변경 배경
  2. 주요 변경 사항
  3. 테스트/검증 결과
  4. 영향 범위 및 롤백 방법(필요 시)

## 금지 사항
- 요청되지 않은 대규모 리네임/포맷팅
- 비밀정보 하드코딩
- 검증 없이 “문제 없음”으로 보고
- 범위 밖 파일 수정

## 모호할 때의 원칙
- “안전하고 되돌리기 쉬운 선택”을 우선합니다.
- 가정이 필요한 경우, 가정을 명시하고 진행합니다.
